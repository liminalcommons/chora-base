### SAP Structure Decision

**Question**: "Should these be new SAPs (014: Coordination Patterns, 015: Commons Philosophy) OR modify existing SAPs (extend 001/009/010) OR hybrid approach?"

**Response**: ✅ **Recommend Option A (New SAPs) with modifications**

**Reasoning**:

#### Why New SAPs (Not Modify Existing)

**SAP-001 (Inbox Protocol)**:
- Currently focused on **request/response mechanics** (schemas, triaging, processing)
- Adding ecosystem coordination + philosophy would conflate **tool** (inbox) with **approach** (coordination philosophy)
- Keep SAP-001 as technical protocol; reference SAP-014/015 for ecosystem patterns

**SAP-009 (Agent Awareness)**:
- Currently focused on **context loading, memory integration, progressive phases**
- Philosophy belongs at ecosystem level, not agent implementation level
- Keep SAP-009 as technical awareness patterns; reference SAP-015 for collaboration philosophy

**SAP-010 (A-MEM Memory System)**:
- Currently focused on **memory structure, feedback loops, knowledge notes**
- Cross-repo event correlation could extend SAP-010 BUT better as SAP-014 pattern
- Keep SAP-010 as memory system; SAP-014 references it for coordination memory

**Conclusion**: Modifying existing SAPs would create scope creep and muddy their focused purposes.

#### Why Two SAPs (Not One)

**SAP-014 (Ecosystem Coordination Patterns)**: Technical/procedural
- Capability-based routing
- Event-driven traceability
- Progressive coordination intake
- ROI measurement as learning

**SAP-015 (Commons Philosophy & Symbiotic Intelligence)**: Conceptual/values
- Kin-based collaboration principles
- Bounded rationality communication
- Human-AI symbiosis
- Generative contexts

**Reasoning**:
- Different adoption paths: Some repos may adopt technical patterns (SAP-014) without full philosophy (SAP-015)
- Different skill types: Technical (routing, events) vs Relational (communication, collaboration)
- Allows progressive adoption: Start with SAP-014, deepen with SAP-015 over time

#### Hybrid Consideration

**Could SAP-015 be integrated into SAP-009 (Agent Awareness)?**
- ❌ No: Philosophy is ecosystem-wide, not agent-specific
- ❌ No: Human-AI symbiosis applies to all SAPs, not just agent awareness
- ✅ Instead: All SAPs should eventually reference SAP-015 for tone/communication guidance

**Modified Recommendation**: **Option A with Cross-References**

Create both SAP-014 and SAP-015, but ensure:
1. SAP-014 references SAP-001 (inbox protocol), SAP-010 (memory), SAP-013 (metrics)
2. SAP-015 is referenced by *all* existing SAPs for tone/communication philosophy
3. Future SAPs explicitly cite SAP-015 for collaboration principles

This creates a **philosophy foundation layer** (SAP-015) that all other SAPs build upon.

#### What's Missing for Scale

**Current Gaps** (Not addressed by SAP-014/015):

**1. Conflict Resolution Patterns**
- **Gap**: What happens when repos disagree on priorities/approach?
- **Impact**: Scaling from 2 repos (chora-base ↔ chora-workspace) to 5+ repos will surface conflicts
- **Suggestion**: Add to SAP-014 or separate SAP-016 (Conflict Resolution & Consensus)

**2. Capacity-Based Routing**
- **Gap**: How do we route coordination requests when multiple repos could handle them?
- **Impact**: Without capacity awareness, repos may be overloaded or underutilized
- **Suggestion**: Extend CAPABILITIES.yaml to include capacity signals (available/busy/paused)

**3. Async Communication Patterns**
- **Gap**: Most coordination assumes relatively quick response (days-weeks). What about months?
- **Impact**: External contributors may have different availability rhythms
- **Suggestion**: Document async-first patterns in SAP-014 (deadlines flexible, explicit no-response handling)

**4. Ecosystem Health Dashboard**
- **Gap**: No centralized view of ecosystem status (which repos active, what coordinations in flight)
- **Impact**: Hard to see "whole system" as ecosystem grows beyond 5 repos
- **Suggestion**: Extend `inbox-status.py` pattern to ecosystem-level (cross-repo health monitoring)

**5. External Contributor Onboarding**
- **Gap**: Philosophy assumes "we are us" (kin-based). What about true external contributors?
- **Impact**: May need different language/framing for contributors not in Liminal Commons ownership
- **Suggestion**: SAP-015 could include "Translation Guide" for adapting kin language to external context

**6. Dependency Graph Visibility**
- **Gap**: How do we visualize cross-repo dependencies and coordination flow?
- **Impact**: Complex coordinations (like W3 Health Monitoring with 4 repos) hard to track
- **Suggestion**: Tool or visualization showing repo capabilities + active coordinations (network graph)

#### Recommendation: **Proceed with SAP-014/015, Plan for Scale**

**Phase 1** (Now): Formalize SAP-014/015 based on 4-sprint evidence
**Phase 2** (After adoption): Gather feedback from chora-workspace + 1-2 peer repos
**Phase 3** (6 months): Assess gaps 1-6 above based on real scaling experience
**Phase 4** (v4.2-v5.0): Address scaling gaps (new SAPs or extensions)

**Rationale**: Don't let perfect be enemy of good. SAP-014/015 solve immediate coordination needs. Scaling challenges will reveal themselves through practice (emergent aims principle).
